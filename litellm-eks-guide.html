CLUSTER & NETWORKEKS 集群 & 网络配置

1.1 创建 EKS 集群

推荐用 eksctl 声明式创建，确保多 AZ 私有子网。

YAML cluster.yaml 
# eksctl create cluster -f cluster.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: litellm-cluster
  region: us-east-1    # 替换为你的 Region
  version: "1.29"

iam:
  withOIDC: true       # 必须：启用 OIDC for IRSA

managedNodeGroups:
  - name: litellm-ng
    instanceTypes: ["m6i.xlarge", "m7i.xlarge"]
    minSize: 3
    maxSize: 20
    desiredCapacity: 3
    privateNetworking: true   # 节点在私有子网
    amiFamily: AmazonLinux2
    iam:
      withAddonPolicies:
        awsLoadBalancerController: true
        cloudWatch: true
    volumeEncrypted: true
    volumeSize: 100

addons:
  - name: vpc-cni
    version: latest
    configurationValues: '{"enableNetworkPolicy":"true"}'  # 启用 NetworkPolicy
  - name: coredns
  - name: kube-proxy
  - name: aws-ebs-csi-driver

2.4 Karpenter（节点自动扩容）

BASH install-karpenter.sh 
# 设置变量
export CLUSTER_NAME=litellm-cluster
export KARPENTER_VERSION=1.3.0
export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
export REGION=us-east-1
export CLUSTER_ENDPOINT=$(aws eks describe-cluster --name $CLUSTER_NAME --region $REGION --query "cluster.endpoint" --output text)
export CLUSTER_SG=$(aws eks describe-cluster --name $CLUSTER_NAME --region $REGION --query "cluster.resourcesVpcConfig.clusterSecurityGroupId" --output text)
export VPC_ID=$(aws eks describe-cluster --name $CLUSTER_NAME --region $REGION --query "cluster.resourcesVpcConfig.vpcId" --output text)

# 1. 给 private 子网和集群安全组打 Karpenter 发现标签
PRIVATE_SUBNETS=$(aws ec2 describe-subnets \
  --filters "Name=vpc-id,Values=$VPC_ID" "Name=tag:kubernetes.io/role/internal-elb,Values=1" \
  --query "Subnets[].SubnetId" --output text --region $REGION)
for subnet in $PRIVATE_SUBNETS; do
  aws ec2 create-tags --resources $subnet --tags Key=karpenter.sh/discovery,Value=$CLUSTER_NAME --region $REGION
done
aws ec2 create-tags --resources $CLUSTER_SG --tags Key=karpenter.sh/discovery,Value=$CLUSTER_NAME --region $REGION

# 2. 创建 Karpenter Node IAM 角色
aws iam create-role --role-name KarpenterNodeRole-$CLUSTER_NAME \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}'
for policy in AmazonEKSWorkerNodePolicy AmazonEKS_CNI_Policy AmazonEC2ContainerRegistryReadOnly AmazonSSMManagedInstanceCore; do
  aws iam attach-role-policy --role-name KarpenterNodeRole-$CLUSTER_NAME --policy-arn arn:aws:iam::aws:policy/$policy
done
aws eks create-access-entry --cluster-name $CLUSTER_NAME \
  --principal-arn arn:aws:iam::${AWS_ACCOUNT_ID}:role/KarpenterNodeRole-$CLUSTER_NAME \
  --type EC2_LINUX --region $REGION

# 3. 创建 Karpenter Controller IAM 角色 (IRSA) — 参考 iam-policy.json

# 4. 创建 EC2 Spot Service-Linked Role（首次使用 Spot 时需要）
aws iam create-service-linked-role --aws-service-name spot.amazonaws.com

# 5. Helm 安装 Karpenter
helm upgrade --install karpenter oci://public.ecr.aws/karpenter/karpenter \
  --version ${KARPENTER_VERSION} \
  --namespace kube-system \
  --set "settings.clusterName=${CLUSTER_NAME}" \
  --set "settings.clusterEndpoint=${CLUSTER_ENDPOINT}" \
  --set "serviceAccount.annotations.eks\.amazonaws\.com/role-arn=arn:aws:iam::${AWS_ACCOUNT_ID}:role/KarpenterControllerRole-${CLUSTER_NAME}" \
  --set controller.resources.requests.cpu=1 \
  --set controller.resources.requests.memory=1Gi \
  --set controller.resources.limits.cpu=1 \
  --set controller.resources.limits.memory=1Gi \
  --wait

# 6. 应用 NodePool 配置
kubectl apply -f litellm-eks/11-karpenter-nodepool.yaml
Phase 3

SECRETS MANAGEMENTAWS Secrets Manager 配置

🔒 核心安全原则：任何 API Key 都不能以明文出现在 YAML 文件或 Git 仓库中。所有密钥存储在 AWS Secrets Manager，通过 IRSA（无凭证）注入，并由 External Secrets Operator 自动同步。

3.1 在 Secrets Manager 中创建密钥

BASH create-secrets.sh 
# LiteLLM Master Key（用于保护 Admin API 接口）
aws secretsmanager create-secret \
  --name litellm/master-key \
  --secret-string "sk-litellm-$(openssl rand -hex 32)"

# Salt Key（用于加密存储的密钥）
aws secretsmanager create-secret \
  --name litellm/salt-key \
  --secret-string "$(openssl rand -hex 32)"

# Aurora PostgreSQL 连接串
aws secretsmanager create-secret \
  --name litellm/database-url \
  --secret-string "postgresql://litellm:PASSWORD@aurora-cluster.cluster-xxx.us-east-1.rds.amazonaws.com:5432/litellm"

# ElastiCache Redis
aws secretsmanager create-secret \
  --name litellm/redis-host \
  --secret-string "litellm-redis.xxxxx.0001.use1.cache.amazonaws.com"

# Azure OpenAI（按需）
aws secretsmanager create-secret \
  --name litellm/azure-api-key \
  --secret-string "your-azure-api-key"

aws secretsmanager create-secret \
  --name litellm/azure-api-base \
  --secret-string "https://your-resource.openai.azure.com/"

4.1 Namespace（含 Pod Security Standards）

YAML 00-namespace.yaml 
apiVersion: v1
kind: Namespace
metadata:
  name: litellm
  labels:
    # 强制 restricted 安全策略：禁止 root、特权容器等
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted

4.2 ConfigMap（LiteLLM 模型配置）

YAML 04-configmap.yaml（关键片段） 
litellm_settings:
  # Redis 缓存：相同请求直接返回缓存，减少 LLM API 调用费用
  cache: true
  cache_params:
    type: redis
    host: os.environ/REDIS_HOST     # 从 Secret 注入
    port: "6379"
    password: os.environ/REDIS_PASSWORD
    ssl: true
    ttl: 600

router_settings:
  routing_strategy: least-busy      # 多模型时：优先选负载最低
  num_retries: 3
  fallbacks:
    - {"gpt-4o": ["claude-3-5-sonnet"]}   # 自动降级

4.3 Deployment（完整安全加固版）

YAML 05-deployment.yaml（关键片段） 
# Pod 级别安全上下文
securityContext:
  runAsNonRoot: true
  runAsUser: 1000          # 非 root 用户
  runAsGroup: 1000
  seccompProfile:
    type: RuntimeDefault   # 限制系统调用

# 容器级别安全上下文
securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true   # 只读根文件系统
  capabilities:
    drop: [ALL]                  # 丢弃所有 Linux Capabilities

# 资源限制：requests 保证，limits 防止 OOM
resources:
  requests:
    cpu: "2000m"           # 2 vCPU 保证，对应 num_workers=2
    memory: "4Gi"
  limits:
    cpu: "4000m"           # 4 vCPU 突发上限
    memory: "8Gi"

# 反亲和：强制不同节点，软性偏好不同 AZ
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - topologyKey: kubernetes.io/hostname

# 拓扑分散：均匀分布到所有 AZ
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule

Phase 5

AUTO SCALING弹性扩缩容

HPA — Pod 水平扩缩
  • CPU 利用率 > 65% → 触发扩容
  • Memory 利用率 > 75% → 触发扩容
  • scaleUp 稳定窗口 60s，scaleDown 300s
  • minReplicas: 3（保证多 AZ），maxReplicas: 20
Karpenter — 节点自动供给
  • Pod Pending → 自动创建新节点（~60s 就绪）
  • 支持 On-Demand + Spot 混合（节省成本）
  • WhenEmptyOrUnderutilized 策略自动整合空闲节点
  • 多 AZ 均衡：us-east-1a/b/c
  • Karpenter v1.3.0，部署在 kube-system 命名空间

SECURITY HARDENING安全加固

🔒 纵深防御原则：每一层都有独立的安全控制，单点突破不会导致全局失陷。
🔑 凭证管理
  • IRSA：Pod 通过 K8s ServiceAccount 获取 IAM 临时凭证，无需硬编码 Access Key
  • External Secrets Operator：从 Secrets Manager 自动同步，1 小时轮转
  • Git 仓库：YAML 文件中 零明文密钥
  • LITELLM_MASTER_KEY + LITELLM_SALT_KEY 双重加密用户 Token
🛡 网络安全
  • NetworkPolicy 零信任：默认拒绝所有，白名单放行
  • WAF：OWASP Top 10 规则 + IP Rate Limiting
  • ALB：HTTPS 强制，HTTP → HTTPS 重定向，TLS 1.2+
  • VPC Endpoint：Bedrock/Secrets Manager 不经公网
  • Pod 运行在私有子网，无公网 IP
📦 容器安全
  • Pod Security Standards: restricted（最严格策略）
  • runAsNonRoot: true，UID=1000
  • readOnlyRootFilesystem: true
  • allowPrivilegeEscalation: false
  • capabilities.drop: [ALL]
  • Seccomp Profile: RuntimeDefault
🔐 数据安全
  • EBS 卷加密（Karpenter NodeClass 配置）
  • ElastiCache Redis：in-transit 加密（TLS）
  • RDS Aurora：存储加密 + TLS 连接
  • ALB 访问日志 → S3（审计用）
  • CloudWatch Logs 保存 LiteLLM 应用日志

WAF 规则配置（已部署）

WAFv2 WebACL litellm-waf 已关联到 ALB，通过 Ingress 注解 alb.ingress.kubernetes.io/wafv2-acl-arn 绑定。
规则优先级模式说明
AWS Managed Core Rule Set1Count（仅记录）SQL 注入、XSS、路径遍历等 OWASP Top 10
AWS Managed Known Bad Inputs2Count（仅记录）已知恶意请求模式
Rate Limit 1000/5min per IP3Block（拦截）单 IP 每 5 分钟超 1000 次请求则拦截
BASH 创建 WAF WebACL 
# 创建 WAFv2 WebACL
aws wafv2 create-web-acl --name litellm-waf --scope REGIONAL \
  --default-action Allow={} \
  --rules '[
    {"Name":"AWS-AWSManagedRulesCommonRuleSet","Priority":1,
     "OverrideAction":{"Count":{}},
     "Statement":{"ManagedRuleGroupStatement":{"VendorName":"AWS","Name":"AWSManagedRulesCommonRuleSet"}},
     "VisibilityConfig":{"SampledRequestsEnabled":true,"CloudWatchMetricsEnabled":true,"MetricName":"CommonRuleSet"}},
    {"Name":"AWS-AWSManagedRulesKnownBadInputsRuleSet","Priority":2,
     "OverrideAction":{"Count":{}},
     "Statement":{"ManagedRuleGroupStatement":{"VendorName":"AWS","Name":"AWSManagedRulesKnownBadInputsRuleSet"}},
     "VisibilityConfig":{"SampledRequestsEnabled":true,"CloudWatchMetricsEnabled":true,"MetricName":"KnownBadInputs"}},
    {"Name":"RateLimit-1000-per-IP","Priority":3,
     "Action":{"Block":{}},
     "Statement":{"RateBasedStatement":{"Limit":1000,"EvaluationWindowSec":300,"AggregateKeyType":"IP"}},
     "VisibilityConfig":{"SampledRequestsEnabled":true,"CloudWatchMetricsEnabled":true,"MetricName":"RateLimit"}}
  ]' \
  --visibility-config SampledRequestsEnabled=true,CloudWatchMetricsEnabled=true,MetricName=litellm-waf \
  --tags Key=Project,Value=litellm --region us-east-1

# 将返回的 ARN 填入 07-ingress.yaml 的 wafv2-acl-arn 注解后 apply
kubectl apply -f litellm-eks/07-ingress.yaml

# 观察 Count 指标，确认无误报后切换为 Block：
# aws wafv2 update-web-acl ... OverrideAction → {"None": {}}

Redis 缓存效益

YAML config.yaml — 缓存配置 
litellm_settings:
  cache: true
  cache_params:
    type: redis
    host: os.environ/REDIS_HOST
    port: "6379"
    ssl: true
    ttl: 600                   # 缓存 10 分钟
    # 只缓存完整响应（非流式），流式请求不缓存
    supported_call_types:
      - acompletion
      - completion
      - embedding
✅ 生产环境实测：开启 Redis 缓存后，重复 prompt 的响应时间从 2-5s 降至 <10ms，同时直接节省 LLM API 调用费用。

SCALABILITY扩展策略

HPA（水平扩 Pod）
CPU > 65% 或 Memory > 75% 时，每 60s 最多扩 4 个 Pod 或当前数量的 50%，取较大值。缩容保守：等待 5 分钟稳定窗口。
Karpenter（扩节点）
Pod Pending 时自动 60s 内创建新节点。支持 On-Demand + Spot 混合。节点空闲 60s 后自动合并回收，降低成本。
PDB（高可用保障）
minAvailable=2：任何时刻（包括节点维护、滚动更新）至少 2 个 Pod 在运行，保证服务不中断。
